create agents, configure them, connect them into multi-agent workflows, and interact

## Design System — Stitch MCP Integration

### What Stitch Produces
The `DESIGN.md` file at the project root is generated by Google Stitch via the
`@_davideast/stitch-mcp` MCP server. It contains:
- Color tokens (exact hex values, CSS variable names)
- Typography scale (font families, sizes, weights, line heights)
- Spacing scale
- Component patterns (card, button, input, badge variants)
- Layout rules (sidebar width, header height, grid columns)

### Rules for Claude Code

1. **ALWAYS read `DESIGN.md` before writing any frontend component.** The design
   tokens in that file override any defaults you would otherwise choose.

2. **Use the CSS variable names from `DESIGN.md` verbatim.** Do not invent new
   variable names. If a token doesn't exist for something, use the closest match
   and add a comment `// TODO: add token to DESIGN.md`.

3. **Never hardcode colors, fonts, or spacing.** Every visual value must trace back
   to a token in `DESIGN.md` or a Tailwind config value derived from it.

4. **When adding a new screen or component**, start by asking yourself:
   "Which Stitch component pattern from `DESIGN.md` is closest to what I need?"
   Extend that pattern rather than creating from scratch.

5. **Motion and micro-interactions** should match the tone established in `DESIGN.md`.
   If the design system is refined/minimal, use subtle transitions (150–200ms ease).
   If it's expressive/bold, use intentional animations (300ms with easing curves).

6. **The ReactFlow canvas** is the hero component of this UI. It must feel premium:
   - Custom node components that use DESIGN.md tokens for colors and typography
   - Animated edges (stroke-dasharray pulse) when a workflow is running
   - Node highlight (glow or ring) on the currently-executing agent
   - Dark-mode first unless DESIGN.md specifies otherwise

### How to Refresh the Design System
If the Stitch MCP is connected in Claude Code:
```
Use the Stitch MCP to fetch the latest design for the agent platform dashboard.
Export the DESIGN.md and overwrite the existing one.
Then update frontend/tailwind.config.ts to sync the new tokens.
```

## Architecture Rules

### 1. Layer Separation (hard rules)
- `backend/api/` — HTTP routing only. No business logic. No LangGraph imports.
- `backend/core/` — All LangGraph graph construction. No FastAPI imports.
- `backend/persistence/` — All DB access. No agent logic.
- `backend/streaming/` — SSE serialization only. Pure functions.
- `frontend/` — Zero business logic. Fetch from API, render, stream.

**Never import across forbidden boundaries.** If you feel the urge to put graph
logic in a router, create a service layer in `backend/core/` instead.

### 2. Agent Graph Pattern

Always use the **Supervisor pattern** via `langgraph-supervisor`:

```python
from langgraph_supervisor import create_supervisor
from langchain.agents import create_agent
from langgraph.checkpoint.sqlite import SqliteSaver

# Workers are stateless — state lives in the graph checkpointer
researcher = create_agent(model=llm_mini, tools=[web_search], name="researcher",
                          prompt="You are a research agent...")
writer     = create_agent(model=llm_mini, tools=[],            name="writer",
                          prompt="You are a writing agent...")

workflow = create_supervisor(
    [researcher, writer],
    model=llm,
    prompt="Route tasks to the appropriate specialist agent.")

app = workflow.compile(checkpointer=SqliteSaver.from_conn_string("agents.db"))
```

Every workflow invocation MUST pass a `thread_id` in config:
```python
config = {"configurable": {"thread_id": thread_id}}
result = await app.ainvoke(input, config=config)
```

### 3. Streaming — Always SSE, Never Polling

Use `stream_mode=["updates", "messages", "custom"]` everywhere:

```python
async for chunk in graph.astream(input, config=config,
                                  stream_mode=["updates","messages","custom"],
                                  version="v2", subgraphs=True):
    yield f"event: {chunk['type']}\ndata: {json.dumps(serialize(chunk))}\n\n"
```

Always set these headers on SSE responses:
```python
headers = {"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
```

**Never use polling.** If something needs live updates, it gets an SSE endpoint.

### 4. Persistence Rules

- Use `SqliteSaver` for LangGraph checkpoints (conversation memory per thread).
- Use a small `runs` SQLAlchemy table for run metadata (start_time, status, tokens, cost).
- Use a `agents` table for CRUD-managed agent configs (name, role, system_prompt, model, tools).
- Database file: `agents.db` in project root, mounted as a Docker volume.
- **Never use Postgres** in this project — SQLite is deliberate (zero infra, single setup command).

### 5. Telegram Integration

```python
# aiogram webhook mounts on FastAPI — single process, single event loop
@tg_router.message()
async def on_telegram_message(msg: Message):
    thread_id = f"tg-{msg.chat.id}"
    config = {"configurable": {"thread_id": thread_id}}
    # Stream tokens back, throttle edits to max 1/second (Telegram rate limit)
    placeholder = await msg.answer("Thinking…")
    buf = ""
    async for chunk in graph.astream(..., config=config, stream_mode=["messages"]):
        if chunk["type"] == "messages":
            tok, meta = chunk["data"]
            buf += tok.content or ""
            if len(buf) % 100 == 0:   # throttle
                await placeholder.edit_text(buf or "…")
    await placeholder.edit_text(buf)
```

The `thread_id = f"tg-{chat_id}"` pattern means:
- Every Telegram user has isolated conversation history
- The SAME conversation can be continued in the web UI by using the same `thread_id`
- This is a demo showstopper — always make sure this works

## Code Style Rules

### Python (backend)
- Type every function signature. `Any` is forbidden.
- Use `pydantic-settings` for all config — no `os.getenv()` calls outside of `config.py`.
- Use `structlog` for logging — never `print()`.
- Every FastAPI endpoint returns a typed Pydantic response model.
- Async everywhere in the agent/streaming layer. `def` routes are allowed only for
  pure CPU-bound sync operations (rare).
- Error handling: wrap LangGraph calls in try/except, return structured error events
  over the SSE stream rather than closing the connection.

```python
# Good
@router.post("/agents", response_model=AgentResponse, status_code=201)
async def create_agent(body: CreateAgentRequest, db: AsyncSession = Depends(get_db)):
    ...

# Bad
@router.post("/agents")
def create_agent(body: dict):
    ...
```

### TypeScript (frontend)
- No `any`. Use `unknown` + type guards when necessary.
- All API calls go through `lib/api.ts` — no raw `fetch` calls in components.
- All SSE subscriptions go through `lib/useAgentStream.ts` — no raw `EventSource` in components.
- Zustand stores are typed with `StoreApi<T>`.
- Components are named exports, pages are default exports.

### Git commits
- Conventional commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`
- One logical change per commit
- Never commit `.env`, `agents.db`, or `node_modules`

## LangGraph Framework Justification (for README)

> Copy this into the README verbatim and edit the tone to match yours.

"We chose LangGraph 1.1 over CrewAI and AutoGen for three reasons specific to this
project's requirements:

**Stability.** LangGraph 1.0 was the first stable major release in the durable agent
framework space (October 2025). For a platform that must run reliably and be reasoned
about in a live code review, a GA framework is the right foundation.

**Observability.** LangGraph's `stream_mode=["updates","messages","custom"]` gives us
per-node, per-token streaming with agent metadata attached — exactly what we need to
drive the live canvas and monitoring UI. No other framework exposes this at the same
granularity without custom instrumentation.

**Durable state.** The SQLite checkpointer gives every conversation thread persistent,
resumable state. This is what allows a Telegram conversation to continue seamlessly
in the web UI using the same `thread_id` — not a demo trick, but a direct consequence
of LangGraph's architecture.

CrewAI is faster to bootstrap but its debugging experience is weaker for complex cyclic
flows. AutoGen is officially in maintenance mode as Microsoft consolidates around its
new Agent Framework (public preview October 2025), making it the wrong bet for a
production-ready platform pitched today."

## Common Pitfalls — Read Before You Get Stuck

| Symptom | Cause | Fix |
|---|---|---|
| SSE stream arrives all at once at the end | Proxy buffering | Add `X-Accel-Buffering: no` header |
| Telegram stops responding after burst | Rate limit (>1 msg/sec) | Throttle `edit_text` calls |
| `langgraph.prebuilt` import error | Module moved in LangGraph 1.0 | Use `langchain.agents.create_agent` |
| `reactflow` import not found | Package renamed | Use `@xyflow/react` |
| SqliteSaver autocommit error | Only affects Postgres saver | Irrelevant for SQLite — ignore |
| Token cost undercounted for o1/o3 | Reasoning tokens not included | Sum `output_token_details.reasoning_tokens` manually |
| Canvas nodes not highlighting | `thread_id` mismatch between SSE and invoke | Use the same `thread_id` in both calls |
| Checkpoint not found on web UI after Telegram | `thread_id` format differs | Always use `f"tg-{chat_id}"` from both entry points |
